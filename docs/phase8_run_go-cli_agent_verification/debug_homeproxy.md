# 核心结论

这份文档的价值在于：

> 它定义了 HomeProxy 的完整“状态真相模型”（State Truth Model）。

它告诉你：

* 系统状态在哪里
* 哪些是持久状态
* 哪些是运行时状态
* 哪些是派生文件
* 如何按层排查问题

---

# 开发者真正需要知道的

## 1️⃣ 唯一事实来源分层

HomeProxy 的状态分为四层：

1. **UCI**（持久配置源）
2. **HP_DIR**（生成逻辑与资源）
3. **RUN_DIR**（当前运行时产物）
4. **procd / nft / cron**（外部执行状态）

任何“系统当前在做什么”的问题，都必须落到这四层之一。

---

## 2️⃣ 所有运行行为都是“生成出来的”

* UCI ≠ 实际运行状态
* 运行状态来自：

  * generate_* 脚本
  * firewall 脚本
  * init 启动流程

如果 JSON / nft / 进程不存在，
系统就没有真正运行。

---

## 3️⃣ 排错必须按层进行

排查顺序应为：

1. UCI 是否正确
2. 生成脚本是否成功执行
3. sing-box JSON 是否生成并校验通过
4. procd 是否成功拉起进程
5. nft / ip rule 是否加载
6. 日志是否反映异常

避免“凭感觉排错”。

---

## 4️⃣ UI 不等于真实状态

LuCI 显示“已开启” ≠ 系统真的在跑。

真实状态必须通过：

* 文件是否生成
* 进程是否存在
* 规则是否加载
* 日志是否正常

来验证。

---

## 5️⃣ 这不是教程，而是状态边界定义

这份文档的意义在于：

* 定义状态边界
* 消除幽灵状态
* 防止认知偏差
* 建立可验证模型

它把 HomeProxy 从“脚本集合”
变成了“可推理的状态机”。

---
