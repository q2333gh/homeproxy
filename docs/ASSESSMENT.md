# HomeProxy 技术评估：现代性、成熟度与能力

本文档基于代码与架构，对 **luci-app-homeproxy** 的“是否足够先进/现代”以及“有多成熟、多强大”做客观评估。结论为个人技术判断，供选型与改进参考。

---

## 一、是否足够先进和现代？

### 1.1 结论概览

**整体：在 OpenWrt 生态里属于偏现代、技术栈较新的一档**；和闭源或 x86 上的桌面代理软件比，功能密度和打磨程度有差距，但在路由器场景下已经足够“现代”。

### 1.2 现代之处

| 维度 | 表现 | 说明 |
|------|------|------|
| **核心引擎** | 使用 **sing-box** | 新一代统一代理内核，协议多（VLESS/Reality、Hysteria2、Tuic、Naive 等）， actively 维护，比很多基于 v2ray/xray 或老旧 ss 的方案新。 |
| **防火墙** | **firewall4 + nftables** | OpenWrt 新默认防火墙，nft 替代 iptables，与 fw4 的 include 机制契合；规则按需生成，不依赖旧 iptables-tproxy 链。 |
| **透明代理方式** | **redirect + tproxy + TUN** | 支持三种方式，TUN 可做全栈代理；tproxy 无需改网关，对游戏/UDP 友好；与 sing-box 能力对齐。 |
| **配置层** | **ucode** 生成 JSON | 用 ucode 做配置生成，比纯 shell 可读、可维护性更好；模板化 nft（utpl）便于维护大段规则。 |
| **协议与特性** | 丰富 | Reality、ECH、multiplex、Hysteria2、Tuic、WireGuard 等；订阅解析支持多种 URI 与 SIP008；支持 urltest 选节点。 |
| **自定义模式** | 完整 | 自定义模式下有独立 DNS 服务器/规则、路由节点/规则、规则集，可对接 sing-box 的 rule_set/remote 等，扩展性强。 |
| **安全与隔离** | **ujail + capabilities** | 可选 ujail 跑 sing-box，限制挂载与权限；capabilities 细粒度，比单纯 root 跑更现代。 |

### 1.3 不够“现代”或可再进化的地方

| 维度 | 现状 | 说明 |
|------|------|------|
| **前端** | 传统 LuCI CBI | 仍是 LuCI 经典表单 + RPC，没有 SPA、没有实时连接状态/流量图等“桌面级”体验；README 已注明订阅页在节点很多时响应慢。 |
| **配置界面** | ACL 嵌在多处 | README TODO 写明“将 ACL 设置移到独立页面”；当前直连/代理/游戏/全局等列表分散在控制面，对新手不友好。 |
| **nft 规则** | 单一大模板 | firewall_post.ut 单文件体积大、分支多，README 标注“Refactor nft rules”；可读性和后续扩展会随功能增加变差。 |
| **平台** | 仅 ARM64/AMD64 | Makefile 限定 LUCI_PKGARCH:=all 但依赖 sing-box/firewall4/kmod-nft-tproxy，实际只能在这些架构上跑，MIPS/ARM 等老设备不在考虑范围，属于刻意取舍而非疏漏。 |

**小结（现代性）**：底层（sing-box、nft、ucode、协议与透明代理方式）足够现代；上层（LuCI 交互、ACL 与 nft 的代码组织）有明确改进空间，整体在路由固件里算“偏新、偏先进”的一类。

## 二、有多成熟、多强大？

### 2.1 结论概览

**成熟度：中上——能稳定支撑家庭透明代理和常见进阶需求，但仍有已知短板和未完成项。**  
**能力强度：在“单机路由器 + 透明代理 + 订阅 + 简单服务端”场景下足够强；复杂多 WAN、策略级可视化、企业级审计等不在设计范围内。**

### 2.2 成熟、可靠的一面

- **配置迁移**：有独立的 `migrate_config.uc`，处理历史 UCI 变更（废弃字段、重命名、块拆解、DNS/routing 语义变更），升级后配置可延续，说明有长期维护意识。
- **启停与错误处理**：init 里先 `sing-box check` 再起进程；生成失败会写日志并 return，不强行起一个坏配置；订阅更新失败会恢复服务并打日志。
- **订阅与节点**：订阅解析支持多种格式与协议；去重、黑名单/白名单、主节点/主 UDP 节点缺失时自动回退到第一个节点或关闭 tproxy，避免“全红”。
- **资源与依赖**：china_ip4/ip6、gfw_list、china_list 有版本检查与更新脚本；依赖明确（sing-box、firewall4、kmod-nft-tproxy、ucode-mod-digest），打包与运行环境清晰。
- **权限与隔离**：ujail、capabilities、sing-box 用户、证书目录权限在脚本里都有考虑，不是“裸 root 跑到底”。
- **运维接口**：RPC 提供日志清理、连通性检测、密钥/证书生成、资源更新、特性检测等，便于 LuCI 和后续自动化。

### 2.3 不够成熟或偏弱的一面

- **已知 TODO（见 README）**：  
  - 订阅页节点多时响应慢；  
  - nft 规则待重构；  
  - ACL 待独立成页。  
  说明作者自己也认为这些是未完成项，而不是“已经完美”。

- **大 nft 模板**：firewall_post.ut 单文件承担所有分流与 ACL 逻辑，条件分支多，对 nft 不熟的人难以局部修改；重构后会更易维护和扩展。（重要性与推荐做法见下文 2.3.1。）

- **可观测性**：无内置流量统计、无连接数/带宽曲线、无按节点/规则维度的统计；日志以文本为主，没有结构化或导出接口。对“只看能不能翻”够用，对“排障与调优”偏弱。

- **多实例/多 WAN**：设计上是单 homeproxy 实例、单套主节点/分流；没有多 WAN 策略绑定、没有“某 WAN 走某节点”的 LuCI 级支持，复杂拓扑需手改或配合其他机制。

- **服务端**：服务端配置生成相对简单（主要是 inbounds），适合个人或小团队自建入口；没有多用户/用量控制/审计等，不属于“企业级网关”定位。

#### 2.3.1 大 nft 模板：有多重要？最佳方案是什么？

**重要程度：中等（可维护性/演进成本，非运行时致命）。**

- **不影响**：当前单文件能正确生成 nft，fw4 加载无问题；日常“只改 UCI、不动脚本”的用户不受影响。
- **影响**：  
  - **改造成本高**：新增分流模式、新增 ACL 类型或新链时，要在 600+ 行、大量 `routing_mode`/`proxy_mode`/`ipv6_support` 分支里找插入点，易漏改或改错。  
  - **协作与冲突**：多人改同一大文件，merge 容易冲突；新人难以“只改某一块”。  
  - **测试与排错**：无法对“仅 DNS 劫持”或“仅 tproxy 链”做小范围验证，只能整份重生成再 fw4 reload。  
  所以这是**长期可维护性与功能演进**的问题，不是“服务起不来”的 P0；但若希望持续加功能、接贡献，值得优先做一次重构。

**推荐方案：按职责拆成“片段 + 单驱动”生成。**

| 做法 | 说明 |
|------|------|
| **1. 按职责拆片段** | 把现有 `firewall_post.ut` 拆成若干**逻辑片段**（仍为 utpl 或 ucode 可渲染的模板），例如：<br>• **sets**：所有 `set` 定义（本机保留地址、gfw/mainland、wan_proxy/wan_direct、routing_port）；<br>• **dns**：DNS 劫持（dstnat 里 53 的 redirect）；<br>• **redirect**：TCP 重定向相关 chain（redirect_proxy、redirect_lanac、redirect 等）；<br>• **tproxy**：UDP tproxy 相关 chain（mangle_prerouting/output + lanac/tproxy）；<br>• **tun**：TUN 模式相关 chain（另一套 mangle + tun_mark）。<br>每块只关心一类规则，条件分支只保留该块需要的（如 redirect 块只看 `proxy_mode` 是否含 redirect）。 |
| **2. 单驱动拼成一份 nft** | 保持 fw4 只 include **一个** `fw4_post.nft`（与当前一致）。用**一个驱动脚本**（如 `firewall_post.uc`）做：<br>• 读 UCI 一次，算出共享变量（`routing_mode`、`proxy_mode`、`control_info`、各 port/mark 等）；<br>• 按顺序调用各片段的渲染（若 utpl 支持 `include` 则主模板 include 各 partial；若不支持，则用 ucode 依次渲染多个小 .ut 或 .uc 片段，把输出拼成一段字符串）；<br>• 将拼接结果写入 `fw4_post.nft`。<br>这样 init 里仍只需调一次“生成 post”的入口（当前是 `utpl -S firewall_post.ut`，可改为 `ucode firewall_post.uc` 或保留主 ut 再 include 各 partial）。 |
| **3. 共享上下文** | 所有片段共用同一套“UCI 解析结果 + 辅助函数”（如 `resolve_mark`、`array_to_nftarr`、`resolve_ipv6`）。可把共享部分抽成 ucode 模块（如 `homeproxy_fw4.uc`），由驱动 require 后传给各片段，避免在每个片段里重复读 UCI。 |
| **4. 可选：按模式再细分** | 若某片段内分支仍很多，可再按 `proxy_mode` 或 `routing_mode` 拆成 redirect_only.ut、tproxy_only.ut、tun_only.ut，驱动按当前模式只渲染需要的组合，减少生成出的 nft 里无用的空链。 |

**落地顺序建议**：先拆成 5 个片段（sets / dns / redirect / tproxy / tun），驱动用“主 ut 里 utpl include 子 ut”或“ucode 顺序渲染 5 个 partial 再拼文件”；验证行为与当前单文件完全一致后，再考虑按模式细分或抽公共模块。这样**不改变 fw4 的用法、不改变生成结果**，只改代码组织，风险小、收益是后续加功能与审阅都更清晰。

### 2.4 能力边界（能做什么 / 不做什么）

| 能做得较好 | 不侧重或不做 |
|------------|-----------------------------|
| 家庭/宿舍透明代理（redirect/tproxy/tun） | 多 WAN / 策略路由级精细控制 |
| 多种分流模式（绕过大陆、GFW、全局、自定义） | 流量/连接数可视化、审计日志 |
| 订阅 + 多协议节点 + 自动更新与回退 | 企业级用户与权限、计费 |
| 在路由器上同时做客户端 + 简单服务端 | 非 ARM64/AMD64 或非 fw4 环境 |
| 自定义模式下的高级 DNS/路由/规则集 | 订阅页在“海量节点”下的高性能 |
| 证书/密钥生成、资源列表更新、迁移兼容 | 复杂 nft 的“可读性/可维护性” |

## 三、综合判断

- **是否足够先进和现代？**  
  **在 OpenWrt/LuCI 生态里：是。** 内核与防火墙选型新、协议与透明代理方式全、配置层用 ucode、有安全与迁移考虑。和桌面/商业代理软件比，UI 与可观测性不算现代，但定位是路由固件插件，这一点可以接受。

- **有多成熟、多强大？**  
  **成熟度：中上。** 能长期用、升级有迁移、错误与回退有考虑；README 的 TODO 和 nft 单文件问题说明尚未到“非常成熟”。  
  **能力：在“家庭/个人透明代理 + 订阅 + 简单服务端”场景下足够强；** 超出这一范围（多 WAN、企业、海量节点 UI、深度可观测）则需预期要自己扩展或接受现状。

- **or not？**  
  **没有“or not”的二元结论。** 若你的需求是：在 ImmortalWrt/OpenWrt（ARM64/AMD64）上做透明代理、用 sing-box、要订阅和多种分流与 ACL，HomeProxy 是**现代且够用**的选择；若你要的是桌面级 UI、企业级策略与审计、或多 WAN 精细策略，它**目前不是**为那些目标设计的，需要看其他方案或接受二次开发。

## 四、建议的改进方向（若追求更成熟、更“现代”）

1. **订阅与节点**：对订阅页做分页或虚拟滚动，或后端分片返回，缓解“大量节点时慢”的问题。  
2. **ACL**：将直连/代理/游戏/全局等列表与 WAN 段收到独立“ACL”页面，并考虑预设模板（如“仅游戏机走代理”）。  
3. **nft**：将 firewall_post 按“DNS / TCP 重定向 / UDP tproxy / TUN / 地址集”拆成多个小片段或子模板，由主模板 include，便于阅读和单独测试。  
4. **可观测性**：至少提供“当前连接数”或“按 tag 的简单计数”（若 sing-box 或内核暴露），或在状态页增加日志级别与条数限制，便于排障。  
5. **文档**：在仓库内保留或引用 PRINCIPLE.md、本 ASSESSMENT.md，方便贡献者和用户判断“能做什么、不能做什么、值不值得上”。

以上评估基于当前代码与 README；随版本迭代，成熟度与短板会变化，建议以最新代码和 Issue/PR 为准。
